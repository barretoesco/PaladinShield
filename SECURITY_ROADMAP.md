# Security roadmap — PaladinShield

## Hackathon transparency (demo / video)

During the competition, an **embedded OpenAI key** may be included in the extension client (`src/extension/scripts/openai-env.js`, generated via `env:sync` from `DEMO_OPENAI_API_KEY`) to **reduce demo friction**.

That choice is **not** the target security model for a public product: anyone with access to the `.crx` package or source can extract the key, abuse quota, or impersonate calls. Before any **release** or broad public distribution, the key must be **removed from the source tree** and rotated at the provider.

Also distinguish the **demo vehicle** from the **product vision**: for hackathon time constraints, PaladinShield ships as a **browser extension** to make the Runtime Enforcement Layer (REL) visible on signing calls. Post-hackathon, the thesis is **not** to remain a standalone extension, but to **embed this REL layer in Solana ecosystem wallets** so protection happens **directly on the user’s signing surface**, without requiring every user to install a separate wrapper.

## Production architecture (orientation)

- **No secrets in the extension host:** the browser sends only necessary metadata (e.g. intent summary, hashes, non-sensitive context) to a **owned backend** that holds the API key or a signed proxy.
- **Authentication and limits:** user session, rate limiting, audit logging, and optionally signed payloads between extension and service.
- **Compliance:** minimal data retention, DPA with the LLM provider where applicable, and secure channels (TLS, CSP headers on embedded UI where relevant).

## Post-hackathon vision

- **Wallet-native REL:** the strategic direction is to integrate the enforcement engine into Solana wallets or expose it as a wallet-integrable layer — not to cap the product at a demo extension.
- **Universal protection at the signing point:** every ecosystem user should get the same semantic control, default-deny, and forensic evidence where they approve transactions or `signMessage`.
- **Ecosystem infrastructure:** long term, PaladinShield should be positioned as **standard security middleware / REL** for Solana products so wallets, apps, and other frontends inherit protection without rebuilding their own review stack.

## Frictionless evaluation (judges / reviewers)

- **REL core (Promise gate + Tier C heuristics)** does not require OpenAI — see [docs/REVIEWER_FAQ.md](docs/REVIEWER_FAQ.md) and [docs/JUDGES_LAB_GUIDE.md](docs/JUDGES_LAB_GUIDE.md).
- **Post-hackathon production:** a temporary reverse proxy (Cloudflare Workers, Vercel, etc.) with rate limits so the unpacked build carries no secrets and judges are not asked for a key — aligned with the backend architecture above.
- **`npm run env:sync`:** for developers only, when testing Tier B semantic enrichment locally.
